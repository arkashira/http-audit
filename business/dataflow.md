# Dataflow Architecture for HTTP Audit

## External Data Sources
- **HTTP Traffic**: Incoming and outgoing HTTP requests and responses from web servers.
- **Kernel Logs**: System logs generated by the Linux kernel, capturing events related to HTTP server interactions.
- **Network Monitoring Tools**: Data from tools like Wireshark or tcpdump for packet analysis.
- **User Inputs**: Configuration settings and parameters from system administrators.

## Ingestion Layer
- **HTTP Traffic Collector**: Captures real-time HTTP traffic data.
- **Kernel Log Parser**: Extracts relevant information from kernel logs.
- **Network Data Ingestor**: Collects data from network monitoring tools.
- **Configuration Interface**: Allows users to input settings and parameters for auditing.

## Processing/Transform Layer
- **Data Normalizer**: Standardizes data formats from different sources.
- **Anomaly Detection Engine**: Analyzes HTTP interactions for unusual patterns or behaviors.
- **Audit Log Generator**: Creates structured audit logs based on processed data.
- **Alerting System**: Triggers alerts for detected anomalies or security incidents.

## Storage Tier
- **Raw Data Storage**: Stores unprocessed HTTP traffic and kernel logs.
- **Processed Data Storage**: Stores normalized and transformed data for querying.
- **Audit Log Database**: A dedicated database for storing generated audit logs.

## Query/Serving Layer
- **Query Interface**: API for querying processed data and audit logs.
- **Dashboard**: User interface for visualizing HTTP interactions and anomalies.
- **Reporting Tool**: Generates reports based on user queries and audit logs.

## Egress to User
- **Notification System**: Sends alerts and notifications to users based on detected anomalies.
- **Export Functionality**: Allows users to export audit logs and reports in various formats (e.g., CSV, JSON).
- **User Authentication**: Ensures secure access to the dashboard and query interface.
